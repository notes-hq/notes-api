# Notes API

Backend-часть информационной системы заметок с тегами. Проект предоставляет REST API для создания, чтения, редактирования, удаления, поиска и фильтрации заметок, а также обеспечивает долговременное хранение данных в MySQL.

## Демо

Рабочая версия проекта доступна по адресу:

```text
http://89.23.116.28/
```

Backend API доступен с префиксом:

```text
http://89.23.116.28/api/v1
```

Демо-стенд размещен на сервере и может быть временно недоступен. Для проверки проекта независимо от сервера используйте локальный запуск по инструкции ниже.

## Назначение проекта

`notes-api` является backend-репозиторием системы заметок. Его основная задача - предоставить стабильный программный интерфейс для работы с заметками, тегами и поиском, отделив хранение данных и бизнес-правила от пользовательского интерфейса.

Backend принимает HTTP-запросы от frontend-приложения, валидирует входные данные, выполняет операции с базой данных и возвращает ответы в едином JSON-формате. Такое разделение позволяет рассматривать backend как независимый модуль системы, к которому в дальнейшем могут подключаться разные клиенты.

## Место backend в общей системе

Проект состоит из двух самостоятельных репозиториев, которые разрабатываются в рамках одного рабочего пространства:

- `notes-api` - backend API на Node.js, Express и MySQL;
- `notes-web` - frontend-приложение на React и Vite.

Такое разделение отражает принцип ответственности компонентов: backend отвечает за данные, бизнес-логику и API-контракт, а frontend - за пользовательский интерфейс и взаимодействие с пользователем.

Ключевое преимущество такой архитектуры состоит в том, что backend предоставляет универсальный REST API. Поэтому к нему можно подключать не только текущий web-интерфейс, но и другие клиенты: консольное приложение, desktop-приложение, мобильное приложение, Telegram-бот или альтернативный web-интерфейс.

## Основные возможности

Backend реализует следующие возможности:

- получение списка заметок;
- получение одной заметки по `id`;
- создание новой заметки;
- редактирование существующей заметки;
- удаление заметки;
- извлечение тегов из `content`;
- нормализация тегов;
- фильтрация заметок по одному тегу;
- поиск по заголовку и содержимому заметки;
- комбинированный поиск по строке и тегу;
- сортировка заметок по дате создания, новые сверху;
- валидация входных данных;
- единый формат ошибок;
- версионирование API через `/api/v1`;
- формальное описание API через `openapi.yaml`.

## Обоснование выбранного стека

Backend реализован на следующем стеке:

- **Node.js** - среда выполнения JavaScript на стороне сервера;
- **Express** - web-фреймворк для построения HTTP API;
- **MySQL** - реляционная система управления базами данных;
- **mysql2** - драйвер для взаимодействия Node.js с MySQL;
- **dotenv** - загрузка переменных окружения;
- **cors** - поддержка запросов от frontend-приложения;
- **ESLint** - проверка единообразия кода;
- **OpenAPI** - формальная спецификация API.

Выбор Node.js и Express позволяет построить компактный, понятный и расширяемый backend. Express хорошо подходит для небольшого REST API: он не перегружает проект лишними абстракциями, но при этом обеспечивает все необходимые возможности для маршрутизации, обработки JSON и построения middleware-цепочек.

MySQL выбран как полноценное долговременное хранилище данных. В отличие от временных или файловых решений, реляционная база данных позволяет явно описать структуру данных, связи между сущностями и ограничения целостности. Для системы заметок с тегами это особенно важно, так как связь между заметками и тегами имеет характер many-to-many.

## Почему MySQL вместо localStorage или JSON-файла

Проект реализован как full-stack система, поэтому постоянное хранение данных вынесено на backend и организовано через MySQL. Это решение осознанно отличается от более простого хранения в `localStorage` или JSON-файле.

Основные причины выбора MySQL:

- **Реляционная модель естественно описывает связь заметок и тегов.** Одна заметка может иметь несколько тегов, а один тег может относиться к нескольким заметкам. Такая связь корректно моделируется через таблицу связей `note_tags`.
- **Целостность данных обеспечивается на уровне базы.** Внешние ключи связывают `notes`, `tags` и `note_tags`, а удаление заметки автоматически очищает связанные записи.
- **Транзакции позволяют сохранять заметку и ее теги как единую операцию.** Если ошибка возникает в середине записи, изменения откатываются.
- **Backend становится универсальным источником данных.** Данные не привязаны к одному браузеру или одному устройству пользователя.
- **Проект готов к дальнейшему расширению.** На основе MySQL проще добавить пользователей, индексы, статистику, более сложную фильтрацию и отдельное управление тегами.
- **Стек соответствует задаче интеграции программных модулей.** Backend, frontend и база данных образуют согласованную систему, а не изолированное клиентское приложение.

Для проекта такого масштаба это решение повышает архитектурную выразительность: система демонстрирует не только работу интерфейса, но и полноценное взаимодействие frontend, backend и базы данных.

## Архитектура backend

Backend построен по слойной архитектуре:

```text
HTTP request
  -> Express app
  -> routes
  -> controllers
  -> services
  -> repositories
  -> MySQL
```

Каждый слой отвечает за свою часть работы:

- `app.js` - настройка Express-приложения, CORS, JSON parser, подключение маршрутов и общий error handler;
- `routes` - описание HTTP-маршрутов;
- `controllers` - прием HTTP-запросов и формирование HTTP-ответов;
- `services` - бизнес-логика, валидация и нормализация данных;
- `repositories` - SQL-запросы и транзакции;
- `config` - подключение к базе данных;
- `utils` - вспомогательные функции, не привязанные к HTTP-слою.

Такое разделение упрощает поддержку проекта. Например, правила валидации можно изменять в service-слое, не затрагивая SQL-запросы, а структуру базы можно развивать в repository-слое, не меняя контроллеры.

## Модель данных

В системе используются три основные таблицы:

- `notes` - заметки;
- `tags` - уникальные имена тегов;
- `note_tags` - таблица связи между заметками и тегами.

### Таблица `notes`

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `INT AUTO_INCREMENT` | Уникальный идентификатор заметки |
| `title` | `VARCHAR(255)` | Заголовок заметки |
| `content` | `TEXT` | Содержимое заметки |
| `created_at` | `DATETIME` | Дата и время создания |
| `updated_at` | `DATETIME` | Дата и время последнего обновления |

### Таблица `tags`

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `INT AUTO_INCREMENT` | Уникальный идентификатор тега |
| `name` | `VARCHAR(100) UNIQUE` | Имя тега без символа `#` |

### Таблица `note_tags`

| Поле | Тип | Назначение |
|---|---|---|
| `note_id` | `INT` | Ссылка на заметку |
| `tag_id` | `INT` | Ссылка на тег |

Первичный ключ таблицы `note_tags` состоит из пары `note_id, tag_id`, что исключает повторную привязку одного и того же тега к одной заметке. Внешние ключи настроены с `ON DELETE CASCADE`, поэтому связанные записи автоматически удаляются при удалении заметки или тега.

## Работа с тегами

Теги извлекаются backend-слоем из поля `content`. Тегом считается отдельный whitespace-разделенный токен следующего вида:

```text
#tag
```

После символа `#` разрешены:

- Unicode-буквы;
- цифры;
- символ подчеркивания `_`.

Максимальная длина имени тега - 100 символов. При сохранении тег:

- очищается от начального `#`;
- приводится к нижнему регистру;
- сохраняется без дубликатов;
- сохраняет порядок первого появления в `content`.

Пример:

```json
{
  "title": "План разработки",
  "content": "Описать backend и frontend\n#study #backend #study"
}
```

Backend извлечет массив тегов:

```json
["study", "backend"]
```

## Валидация данных

Backend выполняет обязательную валидацию входных данных.

### `id`

`id` должен быть положительным целым числом. Некорректные значения приводят к ответу `400 Bad Request`.

### `title`

Заголовок:

- должен быть строкой;
- очищается через `trim`;
- не должен быть пустым после `trim`;
- не должен превышать 255 символов.

### `content`

Содержимое:

- должно быть строкой;
- очищается через `trim`;
- не должно быть пустым после `trim`.

Если в `content` встречается токен, начинающийся с `#`, но не соответствующий формату тега, backend возвращает ошибку `400 Bad Request`.

## Обработка ошибок

API возвращает ошибки в едином формате:

```json
{
  "message": "Error message"
}
```

Для клиентской части важно, что структура ошибки стабильна: объект содержит поле `message`. Дополнительные поля в ошибке не используются.

Основные статусы:

| Статус | Назначение |
|---|---|
| `400 Bad Request` | Некорректный id, пустые поля, слишком длинный title, некорректный формат тега или некорректный JSON |
| `404 Not Found` | Заметка с указанным id не найдена |
| `500 Internal Server Error` | Непредвиденная ошибка backend |

## Работа с датами и UTC

Backend возвращает даты в ISO-формате с суффиксом `Z`, то есть как UTC-время:

```json
{
  "createdAt": "2026-06-07T10:15:00.000Z",
  "updatedAt": "2026-06-07T10:15:00.000Z"
}
```

Для устойчивости настройки времени фиксируются на двух уровнях:

- в параметрах подключения `mysql2` используется `timezone: 'Z'`;
- на каждом новом MySQL-соединении выполняется `SET time_zone = '+00:00'`.

Это снижает зависимость результата от локальных настроек MySQL-сервера и делает формат дат предсказуемым для frontend.

## Транзакции

Создание и обновление заметки выполняются в транзакциях. Это необходимо потому, что операция затрагивает несколько таблиц:

- `notes`;
- `tags`;
- `note_tags`.

При создании заметки backend:

1. Создает запись в `notes`.
2. Создает отсутствующие теги через `INSERT IGNORE`.
3. Получает идентификаторы тегов.
4. Создает связи в `note_tags`.
5. Фиксирует транзакцию.

При обновлении заметки backend:

1. Проверяет существование заметки.
2. Обновляет `title` и `content`.
3. Удаляет старые связи с тегами.
4. Создает новые связи.
5. Фиксирует транзакцию.

Если ошибка возникает до фиксации, транзакция откатывается. Это защищает базу от частично записанных данных.

## Версионирование API

API изначально опубликован с версионным префиксом:

```text
/api/v1
```

Текущие endpoints доступны по адресам вида:

```text
GET /api/v1/notes
POST /api/v1/notes
```

Версионирование добавлено в начальную архитектуру проекта сознательно. Оно позволяет в будущем разработать `/api/v2` с измененным контрактом, не ломая клиентов, которые продолжают использовать `/api/v1`.

В `openapi.yaml` версия вынесена в `servers.url`, а пути описаны относительно этого базового URL. `servers.url` задан как относительный путь `/api/v1`, поэтому Swagger UI использует тот же origin, с которого открыта документация: локально это будет `localhost`, а на сервере - домен или IP сервера. Это соответствует стандартной практике OpenAPI: общий префикс задается один раз, а `paths` содержат ресурсные маршруты.

## Endpoints

| Метод | Путь | Описание | Успех | Ошибки |
|---|---|---|---|---|
| `GET` | `/api/v1/notes` | Список заметок с опциональными фильтрами `search` и `tag` | `200` | `500` |
| `POST` | `/api/v1/notes` | Создание заметки | `201` | `400`, `500` |
| `GET` | `/api/v1/notes/{id}` | Получение одной заметки по id | `200` | `400`, `404`, `500` |
| `PATCH` | `/api/v1/notes/{id}` | Обновление заметки | `200` | `400`, `404`, `500` |
| `DELETE` | `/api/v1/notes/{id}` | Удаление заметки | `204` | `400`, `404`, `500` |

Полная формальная спецификация API находится в [openapi.yaml](./openapi.yaml).

## Query-параметры списка заметок

`GET /api/v1/notes` поддерживает два необязательных параметра:

| Параметр | Назначение |
|---|---|
| `search` | Регистронезависимый поиск подстроки в `title` и `content` |
| `tag` | Фильтр по одному тегу без символа `#` |

Если `search` и `tag` переданы вместе, условия применяются одновременно через логическое `AND`.

Пустые значения или значения только из пробельных символов считаются отсутствующими.

## Примеры запросов

### Получение всех заметок

```http
GET /api/v1/notes
```

### Поиск по тексту

```http

```

### Фильтрация по тегу

```http
GET /api/v1/notes?tag=study
```

### Комбинированный поиск

```http
GET /api/v1/notes?search=express&tag=backend
```

### Создание заметки

```http
POST /api/v1/notes
Content-Type: application/json
```

```json
{
  "title": "Изучить backend",
  "content": "Разобрать Express и MySQL\n#study #backend"
}
```

### Обновление заметки

```http
PATCH /api/v1/notes/1
Content-Type: application/json
```

```json
{
  "title": "Изучить backend API",
  "content": "Разобрать Express, MySQL и OpenAPI\n#study #backend"
}
```

### Удаление заметки

```http
DELETE /api/v1/notes/1
```

Успешное удаление возвращает `204 No Content` без тела ответа.

## Формат успешного ответа

Модель заметки:

```json
{
  "id": 1,
  "title": "Изучить backend",
  "content": "Разобрать Express и MySQL\n#study #backend",
  "tags": ["study", "backend"],
  "createdAt": "2026-06-07T10:15:00.000Z",
  "updatedAt": "2026-06-07T10:15:00.000Z"
}
```

Поле `tags` вычисляется backend на основе `content`, нормализуется и возвращается как массив строк без символа `#`.

## OpenAPI

Файл `openapi.yaml` является формальным источником описания API. Он фиксирует:

- базовый URL API;
- список endpoints;
- параметры запросов;
- форматы request body;
- форматы response body;
- HTTP-статусы;
- схему ошибки;
- схему модели заметки.

README дает обзор API для человека, а `openapi.yaml` содержит строгую спецификацию, которую можно использовать для проверки контракта или подключения внешних клиентов.

## Интерактивная документация API

После запуска backend интерактивная документация доступна по адресу:

```text
http://localhost:3000/api-docs
```

Swagger UI визуализирует существующую спецификацию `openapi.yaml`. Через интерфейс можно просматривать описание endpoints, схемы запросов и ответов, а также отправлять тестовые запросы к backend через кнопку "Try it out".

В спецификации используется относительный server URL `/api/v1`, поэтому интерактивные запросы Swagger UI отправляются на тот же origin, где открыт backend. Это позволяет одной и той же спецификации корректно работать и при локальном запуске, и при размещении за reverse proxy на сервере.

Источником истины API остается файл `openapi.yaml` в корне репозитория. Swagger UI лишь отображает его и не является самостоятельной документацией.

## Переменные окружения

Для запуска в любой среде создайте файл `.env` на основе `.env.example`. Этот файл является единым источником конфигурации backend: его используют сервер, database-скрипты и служебные CLI-утилиты.

```bash
cp .env.example .env
```

В Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

Пример:

```env
PORT=3000
HOST=127.0.0.1
DB_HOST=localhost
DB_PORT=3306
DB_USER=root
DB_PASSWORD=
DB_NAME=notes_db
```

Назначение переменных:

| Переменная | Назначение |
|---|---|
| `PORT` | Порт backend-сервера |
| `HOST` | Host, на котором запускается сервер |
| `DB_HOST` | Адрес MySQL-сервера |
| `DB_PORT` | Порт MySQL |
| `DB_USER` | Пользователь MySQL |
| `DB_PASSWORD` | Пароль MySQL |
| `DB_NAME` | Имя базы данных |

Обычный запуск не требует ручной настройки URL для `seed:demo` и `test:scenario`. Эти утилиты используют тот же `PORT`, что и backend, и автоматически обращаются к локальному API по адресу:

```text
http://127.0.0.1:${PORT}/api/v1
```

Если `PORT=3000`, будет использован `http://127.0.0.1:3000/api/v1`. Если на сервере указан `PORT=3001`, утилиты автоматически будут обращаться к `http://127.0.0.1:3001/api/v1`.

## Создание базы данных

Перед инициализацией таблиц необходимо создать базу данных MySQL:

```sql
CREATE DATABASE notes_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

Имя базы должно соответствовать переменной `DB_NAME` в `.env`.

## Инициализация таблиц

Для создания таблиц выполните:

```bash
npm run init-db
```

Команда выполняет SQL из `database/schema.sql`.

## Очистка данных

Для очистки таблиц и сброса `AUTO_INCREMENT`:

```bash
npm run clear-db
```

Команда предназначена для локальной проверки и повторяемого ручного тестирования.

```bash
npm run seed:demo
```

Утилита создает 20 демонстрационных записей через публичный HTTP API backend. Она не подключается к MySQL напрямую и не обходит валидацию, поэтому при наполнении базы используется тот же путь обработки данных, что и при обычном создании заметки клиентом.

Темы демонстрационных записей охватывают основные аспекты проекта:

- архитектуру backend API;
- разделение frontend и backend;
- реляционную модель данных;
- связь many-to-many для тегов;
- транзакционную целостность;
- версионирование API;
- OpenAPI и Swagger UI;
- React frontend как клиент API;
- конфигурируемый API URL;
- работу с тегами;
- поиск и фильтрацию;
- валидацию входных данных;
- единый формат ошибок;
- проверяемость API;
- сценарную проверку API;
- наполнение демонстрационными данными;
- изначальные ограничения;
- изначально заложенную расширяемость;
- деплой и эксплуатацию на сервере.

Записи создаются в таком порядке, чтобы после стандартной backend-сортировки по новым заметкам сверху интерфейс показывал демонстрационный набор в логической последовательности: от архитектуры backend API к деплою и эксплуатации на сервере.

Команда `seed:demo` не очищает базу данных автоматически. Если нужен полностью предсказуемый демонстрационный набор, сначала выполните:

```bash
npm run clear-db
npm run seed:demo
```

В обычном сценарии дополнительная настройка URL не нужна: утилита использует `PORT` из `.env` и обращается к локальному backend.

Порядок определения URL:

1. `DEMO_API_BASE_URL`, если задан явно.
2. `API_BASE_URL`, если задан явно.
3. Автоматический локальный URL из `HOST` и `PORT`:

```text
http://127.0.0.1:${PORT}/api/v1
```

При отсутствии `PORT` используется `3000`.

```powershell
$env:DEMO_API_BASE_URL = 'http://localhost:3000/api/v1'

```

В Bash-среде это можно сделать одной командой:

```bash
DEMO_API_BASE_URL=http://localhost:3000/api/v1 npm run seed:demo
```

## Связь с frontend

Frontend обращается к backend через переменную окружения `VITE_API_BASE_URL`.

Для локального запуска frontend должен использовать:

```env
VITE_API_BASE_URL=http://localhost:3000/api/v1
```

Внутри frontend-кода версия API не должна прописываться вручную. Клиент добавляет только ресурсный путь, например `/notes`, а полный базовый URL берется из окружения.

12. Удалить заметку через `DELETE /api/v1/notes/{id}`.

Утилита работает как внешний клиент backend API и обращается к приложению через реальные HTTP endpoints. Она не подключается к MySQL напрямую и не изменяет внутреннюю реализацию backend.

Сценарий выполняет следующие действия:

1. Проверяет доступность `GET /api/v1/notes`.
2. Создает несколько тестовых заметок через `POST /api/v1/notes`.
3. Выводит созданные записи в консоль.
4. Проверяет общий список заметок.

Тестовые записи создаются с уникальным маркером запуска и затем удаляются самой утилитой. При ошибке сценарий также выполняет очистку в `finally`, поэтому он не очищает всю базу и не затрагивает данные, созданные пользователем вручную.

## Изначально заложенные ограничения

Текущая реализация системы сфокусирована на полном цикле работы с заметками: создание, чтение, редактирование, удаление, тегирование, поиск и долговременное хранение данных. Поэтому часть возможностей, характерных для промышленных систем, сознательно не включалась в базовый объем проекта. Это позволило сохранить архитектуру понятной, контролируемой и соответствующей объему проекта, не снижая при этом расширяемость системы.

К основным ограничениям текущей версии относятся:

- **Однопользовательская модель работы.** В проекте не реализованы регистрация, авторизация и аутентификация пользователей. Все заметки рассматриваются как принадлежащие одному пользователю или одной рабочей области.
- **Отсутствие ролей и разграничения доступа.** Поскольку система является однопользовательской, в ней не вводились роли администратора, редактора или читателя, а также права доступа к отдельным заметкам.
- **Отсутствие пагинации.** Список заметок загружается целиком. Для текущего объема данных это упрощает контракт API и не требует дополнительных параметров вроде `limit`, `offset` или `cursor`.
- **Один тег в фильтре.** Фильтрация реализована по одному тегу за раз. Multi-select по нескольким тегам не добавлялся, чтобы не усложнять контракт API и пользовательский интерфейс.
- **Простой поиск без морфологии.** Поиск выполняется как регистронезависимый поиск по подстроке в заголовке и тексте заметки. Морфологический поиск, исправление опечаток, полнотекстовые индексы и ранжирование результатов не реализованы, поскольку это отдельная поисковая подсистема, выходящая за рамки базовой CRUD-системы.
- **Отсутствие загрузки файлов и изображений.** Система работает только с текстовыми заметками и тегами. Вложения не включались, так как потребовали бы отдельной модели хранения файлов, настройки загрузки, валидации типов и ограничения размера.
- **Нет отдельного API для управления тегами.** Теги извлекаются из `content` заметки и сохраняются в отдельные таблицы backend-слоем. Отдельные endpoints для создания, переименования или удаления тегов не добавлялись, чтобы сохранить простую модель взаимодействия.

Эти ограничения не являются случайными недоработками. Они фиксируют изначально выбранные границы проекта и отделяют базовую CRUD-систему от более сложной промышленной платформы.

## Изначально заложенная расширяемость

Несмотря на сознательное ограничение функциональности, проект изначально спроектирован не как одноразовый прототип, а как расширяемая система. Архитектурные решения позволяют развивать приложение без полной переработки существующего кода.

К ключевым решениям, обеспечивающим расширяемость, относятся:

- **Версионирование API.** Все endpoints backend доступны через префикс `/api/v1`. Это позволяет в будущем добавить `/api/v2` с измененным контрактом и при этом сохранить работоспособность клиентов, использующих текущий `/api/v1` контракт.
- **Формальная спецификация API.** Контракт backend зафиксирован в `openapi.yaml`. Это упрощает проверку API, документирование, подключение новых клиентов и потенциальную генерацию клиентского кода.
- **Разделение frontend и backend.** Frontend и backend находятся в отдельных репозиториях, но разрабатываются в рамках одного рабочего пространства. Backend предоставляет универсальный REST API, поэтому к нему можно подключать другие клиенты: консольное приложение, desktop-приложение, мобильное приложение, Telegram-бот или другой web-интерфейс.
- **Реляционная модель тегов.** MySQL-схема поддерживает связь many-to-many между заметками и тегами через таблицу `note_tags`. Это создает основу для дальнейшего развития тегов: отдельного управления тегами, статистики, автодополнения и более сложной фильтрации.
- **Слойная архитектура backend.** Backend разделен на маршруты, контроллеры, сервисы и репозитории. Такое разделение позволяет изменять бизнес-логику, работу с базой данных или HTTP-слой независимо друг от друга.
- **Централизованное долговременное хранение.** Хранение данных в MySQL позволяет в дальнейшем добавить пользователей, права доступа, индексы, резервное копирование и более сложные сценарии работы с данными.

`notes-api` реализует самостоятельный backend-модуль системы заметок. Архитектура проекта остается достаточно простой для текущего проектного контекста, но содержит базовые решения, необходимые для дальнейшего развития системы.
